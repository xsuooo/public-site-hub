# 公益站收藏 Goal Card

## 项目与版本

- 项目名称：公益站收藏
- 目标版本：`1.0.0-rc.2` candidate hardening（本地候选，不代表已发布制品）
- 目标状态：G6，本地验证完成；真实双人 RC 验收仍待外部执行

## 目标

- 一句话目标：把当前 Manifest V3 公益/API 站点收藏扩展的加固改动整理为一致的 rc.2 本地候选，并验证到可进入双人 RC 验收的状态。
- 目标用户：两名可信测试者，各自使用独立浏览器 profile、测试账号和测试 Key。
- 核心问题：在多个站点、显式端口、权限拒绝、余额长任务和导入恢复场景下，仍要保持数据、凭据和任务状态正确且可恢复。
- 核心价值：让用户能在本地安全管理站点与 Key，并用可追溯的错误和恢复操作完成余额查询。
- 核心使用场景：收藏或识别站点，按当前 Origin 授权，导入或创建 Key，查询单站或批量余额，停止/继续任务，导入导出并恢复数据。
- 用户最终获得的结果：可加载的扩展包、可复现的验证命令、明确的失败反馈，以及不泄露敏感值的本地数据管理流程。

## 第一版范围

### 必须实现

- 保留站点完整 Origin（含显式端口），保存 URL 时去除 query/hash。
- Popup 显示全部匹配站点并支持健康、分类、标签和关键词筛选。
- Key 与接口地址分离处理；掩码、截断或不可信页面文本不能写入完整 Key。
- HTTPS 权限只在前台用户手势中按 Origin 请求；拒绝一个站点不阻断已授权站点。
- 单站/批量余额刷新具备超时、稳定错误码、协作式停止、继续和重启恢复。
- 导入先预览新增/更新/跳过；替换导入具备恢复快照、恢复和清理能力。
- 脱敏诊断和导出不包含完整 Key、Cookie、真实站点列表、备注或深层 URL。
- 真实扩展 runtime smoke、Playwright UI gate、构建和发布包完整性检查可执行。

### 明确不做

- 仅执行已授权的本地候选提交；不推送、合并、移动标签或发布外部包。
- 不使用生产账号、生产 Key 或真实用户数据。
- 不进行浏览器商店发布、Firefox 适配或自动化真实第三方站点验收。
- 不新增与当前 RC 核心流程无关的产品功能或框架迁移。

## 成功指标与验收标准

- `npm test` 在最终代码状态下通过且无 fail/skip；当前基线为 290/290。
- `node --check popup.js options.js background.js` 通过。
- `npm run build` 和 `npm run verify:package` 通过，包只含运行时白名单文件。
- 可用 Edge/兼容 Chromium 时，`npm run verify:runtime -- --browser=<path>` 和 `npm run verify:ui` 通过；浏览器不可用时记录 BLOCKED，不伪造结果。
- 最终代码状态完成 30 个目标、输入或风险不同的独立测试轮次；本轮 30/30 PASS，失败轮次修复并重验后才计入 PASS。
- 关键敏感信息检查通过：测试输出、诊断、页面文本和发布包中不出现完整 Key/Cookie/query/hash。
- 当前 rc.2 候选与已冻结 `v1.0.0-rc.1` 的差异被明确记录；本地提交前的工作树不得被称为可发布 RC。

## 本地验证结论

- 本地自动化范围已达到可交给双人 RC 验收的门槛。
- 未完成的不是本地代码门禁，而是 Chrome/Edge Stable 的真实权限弹窗、专用账号/Key、五天窗口和两名测试者签署。
- rc.2 本地候选门禁已完成；外部制品仍必须在 `v1.0.0-rc.2` 干净标签 worktree 重新打包，本次不创建标签、不发布。

## 约束与边界

- 技术约束：Manifest V3 classic service worker、`importScripts`、原生 HTML/CSS/JavaScript、`chrome.storage.local`。
- 业务约束：当前 RC 只面向两名可信测试者，真实主链路需专用测试站或本机夹具。
- 安全与隐私：零遥测；只做当前工作区和本地测试环境的安全检查；不写入秘密。
- 外部依赖：Node.js 22+、Edge Stable 或可加载扩展的兼容 Chromium、Playwright；真实账号和人工双人验收不在本次自动化授权内。

## 证据分类

### 已确认事实

- README、RC 文档和 `manifest.json` 定义了上述产品边界。
- `codex/rc2-hardening` 当前基于 `2b601db`，工作树包含已授权的 rc.2 加固修改；`v1.0.0-rc.1` 标签指向更早的冻结提交。
- 当前 `npm test` 为 290/290 通过，递归 JavaScript 语法检查为 61/61。

### 合理推断

- 未提交改动属于 RC 后续加固，重点是备份、维护、权限、并发和 UI 门禁。
- 本轮最有价值的交付是验证并修复这些改动，而不是重新设计产品。

### 临时假设

- 用户授权在当前工作区进行本地实现、依赖安装、构建和测试。
- 可以使用合成数据、本机 NewAPI 夹具和隔离浏览器 profile 做验证。

### 待确认问题

- 两名测试者何时执行真实 Chrome/Edge、专用账号和五天验收。
- 用户已授权将当前加固改动整理为 `rc.2`；标签和外部制品仍不在本次本地提交范围内。

## 主要风险与缓解

- 当前工作树脏且偏离冻结标签：先完成 rc.2 本地提交，再由发布流程在干净标签 worktree 生成制品。
- 浏览器或外部测试站不可用：执行本地逻辑、fixture 和可用浏览器门禁，明确标记真实集成未验证。
- 凭据泄漏：使用 `.invalid`、合成 Key 和脱敏证据；审查日志、诊断和包内容。
- 异步竞态回归：优先运行 balance/storage/import/permission 相关回归，并保留独立测试证据。
