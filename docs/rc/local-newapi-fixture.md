# 本机 NewAPI 测试站

没有可安全联调的外部测试站时，用本机 HTTPS 夹具验证权限、收藏识别、Key 导入/创建、余额刷新以及停止/继续。夹具不使用任何外部账号、生产 Key 或真实余额。

它只是本机联调辅助，不是发布物，也不能证明所有第三方 NewAPI 衍生站完全兼容。

## 安全边界

- 四个站点只监听 `127.0.0.1`，默认 Origin 为 `https://127.0.0.1:41731` 至 `https://127.0.0.1:41734`。
- 假会话、假 Key 和状态只存在 Node.js 进程内存，重启即失效。
- 完整假 Key 只在本机 API 响应中传递；页面、夹具状态和服务日志只显示数量与尾号。
- 证书和私钥生成到系统临时目录，不写入 Git，不会进入 `dist/` 或 RC ZIP。
- 默认不向 Windows 信任库安装证书。只在专用、可删除的 Chrome Profile 内接受本机证书警告。

## 启动

在日常开发工作树运行：

```powershell
npm run rc:fixture
```

启动日志会显示：

- 首个测试站 URL；
- 四个本机 Origin；
- 临时证书路径；
- 证书 SHA-256 指纹。

先在专用 Chrome Profile 打开第一个 URL，核对启动日志中的指纹后，仅对该本机证书继续访问。再打开页面底部的 B、C、D 实例链接。

## 建议联调顺序

1. 在实例 A 点击“登录假账号”。同一 IP 下的实例共享假 Cookie；每个端口的页面会把同一假账号同步到自己的 localStorage。
2. 在首次权限请求中先选择拒绝，确认扩展不收藏、不请求 API；然后从用户手势点击“再次授权”并允许。
3. 收藏实例 A。A 预置一把仅在内存中的假 Key，用于验证掩码列表、按 ID 读取全文和尾号显示。
4. 在实例 B 点击“清空假 Key”和“复位计数”，然后收藏 B。在扩展中发起获取 Key：先取消确认，页面中“API 写入”必须仍为 0；再明确确认，写入与 Key 数量各增加 1。
5. 分别收藏 C 和 D。四个 Origin 带不同显式端口，在存储、打开页面和 `/v1` 调用地址中不得丢失端口。
6. 点击“刷新全部余额”后立即点击“停止”。夹具为四个实例配置了 1.4–2.6 秒的余额延迟：当前站完成后应停止，已完成结果保留，剩余站不记为失败。
7. 再次点击“刷新余额”继续剩余队列，确认不重复处理已完成站。

## 清理

1. 在启动夹具的终端按 `Ctrl+C`。
2. 删除专用 Chrome Profile。
3. 如需清理临时证书文件，删除启动日志显示的整个 `public-site-hub-rc-fixture` 临时目录。证书默认未安装到系统信任库，无需移除系统证书。
